# 菜单读取、定制与预览

## 菜单读取

- 先从 `GET /api/v1/shops/{shop_id}` 读取菜单，再决定如何下单。
- 重点字段：
  - `specs`
  - `attrs`
  - `ingredients`
  - `default_ingredients`

## 默认值与定制

- 如果商品有 `default_ingredients`，优先直接使用。
- `specs` 和 `attrs`：
  - 用户明确指定时，选用户指定值
  - 用户没指定时，默认取第一个可用选项
- 对必须选择但用户没提到的项目，后台直接补默认值，不要把实现细节抛给用户。

## ingredients 结构

`ingredients` 是一个数组，每个元素表示一个分组：

```json
[
  {
    "group_name": "配菜选择",
    "required": true,
    "options": [
      {
        "id": "ig_3_0",
        "name": "默认配菜",
        "price": 0.0,
        "item_id": "811158387116",
        "sku_id": "5672937362399",
        "package_id": "2239839520534084436"
      }
    ],
    "default_id": "ig_3_0"
  }
]
```

- `group_name`: 给用户看的分组名称
- `required`: 是否必选
- `options[].name`: 给用户看的选项名称
- `options[].price`: 该选项相对价格变化
- `options[].id`: 用于互斥判断
- `options[].excludes`: 被选中后不能再选的其他 option id
- `default_id`: 这个分组默认选中的 option id

## 用户可读渲染

- 渲染目标：
  - 标明 `必选` / `可选`
  - 默认值要加粗
  - 有价格变化时标 `+¥X`
  - 不展示 `item_id`、`sku_id`、`package_id`、`excludes`
  - 如果存在互斥或联动，只在最后用一句话提醒

推荐格式：

```text
[商品名]
¥XX，现货

默认做法：
- 规格：**常规份**
- 做法：**默认做法**
- 口味：**标准口味**

可调整项：
- `必选` 规格：**常规份** / 大份
- `必选` 做法：**标准** / 少辣 / 微辣
- `可选` 加料：**默认不加** / 加蛋 `+¥2` / 加肉 `+¥5`

联动提示：
- 部分选项会互相影响，我按默认做法帮你下单最稳；如果你想改，我再按你要的口味调整。
```

## 什么时候要问

- 如果用户没有提出定制需求，并且商品有 `default_ingredients`：
  - 直接使用默认组合，不主动展开所有选项
- 如果用户明确提出需求，例如少辣、不要葱、换配菜、加蛋、去糖、换奶、加风味：
  - 只展开和该需求相关的分组
- 如果商品没有 `default_ingredients`，但有 `required=true` 的 `ingredients`：
  - 先渲染紧凑版菜单，再询问用户
- 如果互斥关系复杂：
  - 不向用户解释技术联动规则，只问最终可选结果

## 商品变体与预览校验

- 同名商品可能混有活动款、套餐、组合装、规格/口味/冷热变体；不要只看名字选第一个。
- 用户只说一个通用商品名时，默认优先理解为：
  - 最基础的单品版本
  - 非强活动绑定款
  - 非多人份
  - 非套餐或组合装
- 选商品时优先级：
  1. 普通单品且最接近用户描述的版本
  2. 普通单品的固定规格/口味/冷热版本
  3. 活动单品版本
  4. 套餐、双份、多人餐、组合装
- `POST /api/v1/orders/preview` 不只是看成功失败；还要检查返回的 `items[].name` 是否符合用户预期。
- 如果用户要的是“基础单品”，但预览结果变成了活动款、套餐、多人份、额外规格版本或 `2 份起购`，就回到菜单里换一个更合适的同类商品重试。
